# Melodialect TODO

作曲アプリとして、生成後の「聴く → 試す → 直す → 比較する → 保存する」を、安全かつ効率よく完結させるためのロードマップ。

## 優先度の定義

- **P0** — データ消失や編集の巻き戻りを防ぐため、他機能より先に必要
- **P1** — 日常的な作曲・編集効率を大きく改善する中核機能
- **P2** — 音質、外部連携、出力品質を改善する機能
- **P3** — 作曲表現と拡張性を広げる機能
- **Later** — 価値はあるが、コア体験の完成後に判断する機能

## 現時点の製品方針

- v1.1までに非破壊編集、SoundFont、主要な編集・出力、ダイアレクト作成GUI、PWA公開を完了した。次期開発は**生成品質と作曲表現の強化**を中心にする
- 初心者の基本操作は **「ダイアレクトを選ぶ → 全体生成 → 再生」** のまま維持し、新しい必須入力や専門用語を増やさない
- エンジン内部の候補生成・評価は高度化しても、その複雑さを設定項目としてそのまま画面へ露出しない
- UIは「基本操作」「日常語による簡単な調整」「折りたたんだ詳細設定」の3層とし、詳細設定を使わなくても完成した曲を生成できるようにする
- ルールベース、固定シードでの再現性、生成根拠の説明可能性を製品の中核として維持する。外部AI APIを次期生成エンジンの前提にしない
- 一般向け配布は引き続き**PWA**を基本とし、公開版の利用者へNode.js、npm、開発環境の導入を要求しない
- 楽曲、設定、ユーザー音源は原則として端末内で処理し、ユーザーが明示的に共有しない限りサーバーへ送信しない
- Electron版はPWAと並行開発せず、大容量音源、SFZ、ファイルシステム、外部MIDIなどPWAの制約が実利用上の問題になった場合に追加する

## v1.2以降 — 生成エンジン強化ロードマップ

この節は、v1.1リリース後に検討した「伴奏作曲、作曲の多様性、コード進行、ベースライン、ダイアレクト」の強化方針を、別セッションから継続できるように記録する。個別機能を増やすだけでなく、**複数候補を作り、破綻と重複を除き、ダイアレクトらしい上位候補から固定シードで選ぶ**共通基盤を先に整える。

### 維持するユーザー体験と設計判断

- [x] 基本フローは「ダイアレクト選択 → 全体生成 → 再生」の3操作以内を維持し、新機能のための必須ダイアログを追加しない
- [x] コード探索、ベース経路評価、セクション別伴奏設計は既定で自動実行し、初心者へアルゴリズムの選択を求めない
- [x] 初心者向けの追加項目は「変化の大きさ」「伴奏の厚さ」「展開」「ベースの動き」など、聴感で理解できる少数の項目に限る
- [x] コード機能、アプローチノート率、声部連結の重みなどは、必要な利用者だけが開く詳細設定へ置く
- [x] 生成根拠は「曲全体の要約 → セクションの意図 → 小節・イベント単位の規則」の3段階で表示し、通常表示を説明過多にしない
- [x] 評価値や探索アルゴリズム名をそのまま表示せず、「サビへ向けて緊張を高めた」「次のAmへ半音で接近した」のように音楽的な意図へ変換する
- [x] 既存の `Song` / `GeneratedSection` / `ChordEvent` / `NoteEvent` を再生・編集・書き出し境界として保ち、内部の探索方式をUIとエクスポートから分離する
- [x] 新しい設定は省略可能な既定値を持たせ、既存プロジェクトとv1ダイアレクトを自動移行または互換読込できるようにする

### P0: 候補生成・評価・再現性の共通基盤

現在はセクションごとの1本のRNGを構成、和声、旋律、伴奏が順番に消費するため、前段の変更が後続パートの結果まで変え得る。最初に乱数と評価の責務を分離する。

- [x] ルートシードから `structure`、`harmony`、`melody`、`bass`、`piano`、`guitar`、`drums`、`humanize` の名前付き乱数ストリームを決定的に派生させる
- [x] セクション番号、パート名、候補番号から独立したサブシードを作り、生成順序の変更に結果が依存しないようにする
- [x] ベースだけ、コードだけ、伴奏だけを再生成した場合、対象外パートがイベント単位で同一になる回帰テストを追加する
- [x] 楽曲・セクションの指紋を定義する。少なくともローマ数字n-gram、ハーモニックリズム、旋律音程輪郭、リズム列、音数密度、音域、ベース輪郭、伴奏の楽器参加・発音位置・ボイシング・展開曲線を含める
- [x] ハード制約の検査を共通化する。イベント重複、不正音価、範囲外、コード被覆不足、空セクション、極端な跳躍などは採用前に棄却する
- [x] 品質指標を共通化する。和声機能の接続、声部移動量、旋律とコードの適合、ベースの滑らかさ、伴奏衝突、セクション対比を段階的に追加する
- [x] 多様性指標を共通化し、同じ進行・リズム・輪郭・伴奏配置へ偏った候補を重複候補として扱う
- [x] 1回の生成で複数候補を内部生成する。候補数は性能計測に基づいて調整できるようにし、固定値をUI仕様にしない
- [x] 単純な最高得点固定ではなく、破綻候補を除外した上位群からシード付き重み選択し、品質を守りながら多様性を残す
- [x] 「安定／標準／冒険」に相当する多様性レベルを内部モデルへ持たせる。基本画面へ出す場合も専門用語を使わない
- [x] 各生成判断に、決定内容、適用規則、狙い、必要に応じて退けた代替案を保持できる provenance（生成根拠）データを持たせる
- [x] 固定シードのスナップショットに加え、複数シードで制約違反率、重複率、ダイアレクト特性を測る統計テストを追加する
- [x] 候補生成を増やしても通常曲の生成時間目標とUI応答性を維持する。必要なら評価処理を段階化またはWorker化する

**完了条件:** 同じ設定とシードで再現でき、パート別再生成が対象外パートへ波及せず、重複・破綻候補を除外した複数候補生成を共通APIから利用できること。初心者の基本操作は変更しない。

### P1: コード進行生成の強化・多様化

現行の定型句と1段階のマルコフ遷移は残しつつ、セクション全体の目的地と曲全体の緊張曲線を扱える生成器へ拡張する。伴奏とベースはコード進行に依存するため、3つの作曲機能ではコード強化を先に実装する。

- [x] コード文字列の内部表現をAST化し、表示・既存JSONとの境界では従来のローマ数字文字列へ変換できるようにする
- [x] ASTで度数、変化記号、品質、テンション、転回形・スラッシュベース、一時的な調的中心を表現する
- [x] セカンダリードミナント、経過ディミニッシュ、借用和音、代理和音、ピボットコード、必要なaltered chordを段階的に扱う
- [x] トニック／前属／属などの和声機能と、到達先・カデンツを持つ機能和声グラフを導入する
- [x] 曲全体と各セクションの緊張曲線から、開始領域、頂点、終止先、ハーモニックリズムを計画する
- [x] Beam Search等で複数の進行候補を生成し、機能的一貫性、声部進行、反復量、ダイアレクト適合度、旋律との適合で評価する
- [x] Verseの進行をChorusで拡張・置換する、反復Chorusで一部だけ変える、Bridgeで対比して主調へ戻すなど、セクション間の和声関係を生成する
- [x] リハーモナイズ候補を、前後のコードだけでなく旋律音、ベースの接続、原案からの変更量で評価する
- [x] 既存の固定、空欄補完、原案復元、コードロックを維持し、新方式でもユーザー原案を暗黙に失わないようにする
- [x] 生成根拠へ、和声機能、緊張の変化、借用・代理の理由、カデンツへの接続を自然文で記録する
- [x] 基本画面の操作は増やさず、詳細設定を使わない場合はダイアレクトとセクション表現から自動計画する

**完了条件:** 局所的に自然なだけでなくセクションの目的地へ向かう進行を複数生成でき、固定コード・リハーモナイズ・自動生成の全経路で説明と再現性が保たれること。

### P1: ベースライン生成の強化・多様化

ベースを伴奏の付属物ではなく独立した作曲パートとして扱い、現在の `bossa` / `melodic` / `drone` と汎用グルーヴを互換プリセットとして新モデルへ移行する。

- [x] `Dialect.groove.bassPattern` から独立した `BassProfile` を定義する
- [x] `BassProfile` に役割（root、pedal、walking、ostinato、counterline）、活動量、音域、シンコペーション、休符、コードトーン率、5度・オクターブ率を持たせる
- [x] 半音・全音アプローチ、経過音、囲い込み、跳躍後の解決、フレーズ末フィルをデータとして記述できるようにする
- [x] 小節ごとの局所選択ではなく、セクション全体の候補経路を動的計画法または同等の探索で選ぶ
- [x] コードの明瞭さ、次コードへの接続、最大跳躍、音域、グルーヴ、反復、メロディとの反行・同方向、キックとの同期を評価する
- [x] セクションごとに役割と活動量を変え、Verseでは簡潔、Chorusでは推進、Bridgeでは対旋律などの展開を作れるようにする
- [x] 初心者向けに露出する場合は「ベースの動き: シンプル／メロディック／活発」程度に集約し、詳細値は折りたたむ
- [x] 生成根拠へ、小節頭のルート、次コードへのアプローチ、ペダル、反行、フィルなどの意図を記録する
- [x] ベースだけの部分再生成、ロック、直接編集、MIDI/WAV/MusicXML出力を既存どおり維持する

**完了条件:** コード進行全体と他パートを参照した複数種類のベースラインを生成でき、ベースだけの再生成で他パートが変わらず、初心者は追加設定なしでダイアレクトらしい結果を得られること。

### P1: 伴奏作曲の強化・多様化

曲全体でピアノ・ギター・ドラムのパターン名を1つ選ぶ方式に加え、セクション、フレーズ、楽器間の役割を計画する `ArrangementPlan` を導入する。自然さや声部連結を改善する「強化」だけでなく、同じコードとメロディから異なる編曲解釈を複数作る「多様化」を明示的な目標にする。

- [x] Intro、Verse、Chorus、Bridge、Outroごとに楽器参加、密度、音域、奏法、フィル、休止を計画する `ArrangementPlan` を定義する
- [x] 同じコードとメロディから、楽器編成、テクスチャ、リズム、ボイシング、密度、セクション展開の複数軸を変えた伴奏候補を生成する
- [x] ピアノ主体で段階的に厚くする案、ギター主体でリズムを前景化する案、楽器を交互に配置して空間を活かす案など、異なる編曲戦略をデータとして定義する
- [x] 楽器ごとの参加小節、発音位置のリズム列、音域、ボイシング幅、小節密度、フィル・ブレイク・合いの手の配置、パート間重複率から伴奏指紋を計算する
- [x] 伴奏指紋が近すぎる候補へ重複ペナルティを与え、パターン名だけが違って聴感上ほぼ同じ候補を並べない
- [x] 無関係なパターンをランダムに混ぜず、ダイアレクトの伴奏語彙、拍子、グルーヴ、セクション規則と互換な範囲で多様化する
- [x] 反復セクションに「同じ／軽い変奏／大きな変奏」を適用し、最終Chorusだけ厚くする等の曲全体の編曲曲線を作る
- [x] メロディ密度が高い場所では伴奏を間引き、長音や休符では内声・合いの手を許可する相互作用を追加する
- [x] ピアノとギターが同じ音域・同じ拍を埋め尽くさないよう、コンピング、アルペジオ、パッド、合いの手などの役割を分担する
- [x] コード間をまたぐ内声とトップノートの声部連結を、単一パターンだけでなく伴奏全体の評価へ広げる
- [x] セクション末のフィル、ブレイク、ピックアップ、コーダをパターン本体から独立したジェスチャーとして生成する
- [x] 拍位置、音価、使用声部、ベロシティ、先取り、適用拍子を記述するデータ駆動の伴奏パターン形式を定義する
- [x] 現在のピアノ・ギター・ドラムパターンを互換プリセットとしてデータ形式へ段階的に移行する
- [x] 初心者向けの調整は「伴奏の厚さ: 控えめ／標準／華やか」「展開: 一定／自然／大きい」程度に集約する
- [x] 生成根拠へ、楽器を加減した理由、セクション対比、フィル、メロディとの役割分担を記録する
- [x] 自動編曲を既定としつつ、従来どおり利用者がピアノ・ギター・ドラムのパターンを明示指定できる経路を残す

**完了条件:** 追加設定なしでもセクションに応じた楽器・密度・音域の展開が生まれること。同じコードとメロディから、楽器編成、リズム、ボイシング、密度、展開のうち複数軸が聴感上異なる伴奏候補を生成でき、各候補がダイアレクトらしさと他パートとの整合性を保つこと。伴奏だけを再生成してもコード、メロディ、ベースは変化せず、従来の手動パターン指定、ロック、編集、出力と両立すること。

### P2: 候補の提示と生成根拠の強化

- [x] 通常の全体生成では自動選択した1案をそのまま開き、候補選択を必須操作にしない
- [x] 必要な利用者向けに「他の候補」を開き、同じ再生位置で少数の差別化された候補をA/Bできるようにする
- [x] 候補カードには数値スコアではなく「コード中心」「リズムを変奏」「ベースが活発」「伴奏編成を変更」など、主な差分を表示する
- [x] 既存のバリエーション履歴、お気に入り、スナップショットと候補を統合し、別々の履歴概念を増やしすぎない
- [x] 曲全体の生成要約を最初に表示し、セクション、小節の順に展開できるようにする
- [x] provenanceから自然文を作る説明層を生成エンジンから分離し、表示文言の変更で生成結果が変わらないようにする
- [x] 説明は適用した事実だけを述べ、評価器が判定していない主観的品質を「最良」などと断定しない

**完了条件:** 初心者は候補UIを開かず従来どおり使え、必要な利用者は候補の音楽的な違いと生成理由を理解して比較できること。

**実装記録 (2026-07-22):** P0〜P2を実装。通常生成は内部3候補、任意の「他の候補」は最大4候補とし、候補数はAPI引数で変更可能。Chromaticの通常曲を3候補生成したローカル計測は平均194.89ms、最大252.16ms（5回）。`test/v12.test.ts` に固定・複数シード、制約、重複、AST、ベース文法、伴奏計画、候補境界の回帰テストを追加した。

### P3: Dialect v2 — ダイアレクトの強化

ダイアレクトを完成品プリセットだけでなく、組み合わせと継承が可能な作曲文法へ拡張する。コード・ベース・伴奏の新モデルが安定してから着手し、先にGUIを新スキーマへ固定しない。

- [ ] ダイアレクトへ `schemaVersion` を追加し、v1定義の検証・読込・v2への移行手順を実装する
- [ ] Harmony lexicon、Cadence grammar、Melody grammar、Bass language、Comping language、Drum groove、Form strategy、Orchestration strategyへ責務を分割する
- [ ] 基底ダイアレクトから一部モジュールを上書きできる継承方式を定義し、同じ定義の重複を減らす

- オーディオ波形編集
- VST／Audio Unitプラグインのホスト
- 本格的なミキシングエフェクトとマスタリング
- DAWと同等のオートメーション編集

## 推奨実装順

1. プロジェクト移行・復元スナップショット
2. 非破壊パラメーター適用と同期A/B比較
3. 画面内ヘルプと推奨値リセット
4. マスター計測と標準SoundFont再生基盤
5. SF2取込、端末内音源ライブラリ、音量・負荷テスト
6. WAVステムとMusicXML出力
7. ピアノロール複数編集とコード編集
8. 歌詞編集、コード先行生成、モチーフ制御
9. ユーザー定義ダイアレクトJSON
10. ダイアレクト作成の基本GUI
11. GitHub Pages公開、PWA、v1.0品質保証と利用案内
12. Electron版などLater項目を利用実績に基づいて再評価
